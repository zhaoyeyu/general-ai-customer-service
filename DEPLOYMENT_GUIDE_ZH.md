# 部署选择指南

## 当前建议

开发阶段继续使用本机，不需要购买云服务器。

模型推理由 OpenRouter 或其他上游服务完成，本机只运行 FastAPI、SQLite、文件解析和 BM25 检索。对于开发调试、录入少量知识文件和一到数名测试者，普通 Windows 电脑足够，不需要 GPU。

当前服务默认只监听 `127.0.0.1:8000`，只有这台电脑能访问。这是现阶段最安全、最省事的方式。

## 四种使用方式

### 1. 仅本机开发

适合现在使用。

```powershell
python run.py
```

优点：零服务器费用、修改方便、API Key 和知识库不离开本机。缺点：电脑关闭后服务停止，其他设备无法访问。

### 2. 同一局域网测试

适合让同一 Wi-Fi 或办公网络内的手机、电脑短期访问。

```powershell
$env:APP_HOST="0.0.0.0"
python run.py
```

其他设备通过 `http://本机局域网IP:8000` 访问。可能需要为 Python 开放 Windows 防火墙的“专用网络”入站访问。

目前管理后台没有登录保护，因此不要在陌生或公共 Wi-Fi 中这样运行，也不要开放路由器端口映射。

### 3. 临时公网演示

适合给异地的少量测试者短时间查看。可以使用 Cloudflare Quick Tunnel 一类的临时隧道，把本机端口映射为随机 HTTPS 地址。

在增加管理后台登录保护之前，不建议执行这一步。临时隧道只适合开发和演示，没有生产 SLA；本机休眠、断网或进程退出都会导致服务不可用。

### 4. 正式云服务器

出现以下任一需求时再购买：

- 真实客户需要全天候访问；
- 需要固定域名、稳定 HTTPS 和持续监控；
- 不希望依赖个人电脑开机和家庭网络；
- 需要多人账号、聊天记录、备份或更高并发；
- 已经有稳定业务流量，而不是偶尔演示。

这个项目的模型计算不在服务器上，初期无需 GPU。少量并发可从 `1 vCPU / 1 GB` 起步，但为了文件解析和系统余量，更建议 `1–2 vCPU / 2 GB RAM`。单实例阶段可以继续用 SQLite；需要多实例或多人同时管理时再迁移 PostgreSQL。

## 推荐路线

1. 现在：本机开发，完成知识库、提示词和真实 Key 调用验证。
2. 内测：先增加管理员登录和访客会话隔离，再用局域网或临时隧道给少量用户试用。
3. 有真实外部用户后：购买靠近主要用户的基础云服务器，配置域名、HTTPS、进程守护、日志、备份和限流。
4. 流量增长后：迁移 PostgreSQL，增加 Redis、异步任务和多实例部署。

## 上云前最低安全清单

- 管理后台必须登录，设置与知识库 API 必须鉴权；
- 客户聊天接口应设置 IP/会话限流和每日额度；
- API Key 只保存在服务端，不能出现在前端代码或响应中；
- 使用 HTTPS，限制上传类型与大小；
- 增加对话隐私说明、日志保留周期和人工转接；
- 定期备份数据库、知识库和本机加密密钥。
